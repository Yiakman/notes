bufferAtCap, root blocks and the merge-demand exemption

Visual walkthrough, grounded in an NTP main chain and an EVM parallel chain — packages/node-sdk/sync

Page

A {from, to} range in a chain's own units. For EVM, block numbers, sized by stepSize.

Root block

One merged ChainBlock per pipeline iteration, stamped by the main chain.

τ (tau)

The current root block's timestamp — what every parallel chain's page must pass.

Buffer & cap

bufferedData holds fetched items; bufferAtCap pauses fetching at the cap.

The pipeline #

Every chain fetches into its own buffer. The merge drains both buffers in lockstep to build one root block at a time.

The runtime then commits the root block to Postgres and runs the state machine over its primitives.

Pages — fixed-width chunks #

genInputRange chains pages back-to-back: each one starts where the last ended, and is stepSize wide.

lastPage.own (the page's to) feeds nextInterval for the next page — that's pagination. A separate value, lastPage.root, is the block's timestamp, used only for merge ordering.

// evm/fetcher.ts — pages are chained by reusing the previous page's edge
intervalFromStart(start) { return { from: start, to: start + stepSize }; }
nextInterval(prevEnd)    { return this.intervalFromStart(prevEnd + 1); }

Root block & τ #

NTP's timestamp is pure arithmetic: block n is always startTime + blockTimeMS × n. Each new root block — and so τ — lands exactly blockTimeMS after the last.

+1,000 ms
block 42τ = 42,000 ms
// ntp/state.ts — τ is born here, straight from the NTP tick
toRootOutput(data) { return { blockNumber: ..., timestamp: this.toRootPage(data), ... }; }
toRootPage(data)   { return toMsTimestamp(data.raw.timestamp); }  // startTime + blockTimeMS·n

A parallel chain (EVM) only gets drained once its own page's timestamp passes τ. Until then, the merge just waits — and not because of what's in its buffer.

Page ≠ data — why the merge waits on the page, not the buffer #

The buffer only holds items that matched something. It can't prove the chain has actually looked at every block up to τ — only the page can prove that. Say EVM has fetched up to block 40 (39,500 ms) and buffered 2 matching events, both ≤ τ=42,000. If the merge drained those 2 events right now, it could be wrong: blocks 41 onward haven't been scanned yet, and a 3rd matching event could still show up in that gap.

So the merge gates on lastPage.root — the upper bound of the most recently fetched page — never on the buffer's size or contents: 

// merge.ts — wait for the PAGE to pass τ, not for data to appear
while (compare(state.lastPage.root, τ) <= 0) {
  yield* state.newPageCondVar.wait();
}
// only now is it safe to drain everything buffered with toRootPage(datum) <= τ

And critically, a page advances even when it finds nothing — updateState updates lastPage unconditionally, while pushing buffer items only happens inside the length check: 

// base/state.ts — pushing items is conditional, advancing the page is not
if (data.output.length > 0) {
  for (const datum of data.output) this.bufferedData.push(datum);
  this.newDataCondVar.wake();
}
this.lastPage = data.lastPage;   // ← happens regardless of whether anything was found
this.newPageCondVar.wake();
"Page ≠ Data. lastPage advances even on an empty query, so a parallel chain can signal 'scanned up to T, found nothing' and let the merge proceed." sync/CLAUDE.md — design idea #2

An empty page is just as valid a completeness proof as a full one — it's the page boundary that unblocks the merge, not the data.

What's actually in the buffer #

Each fetch can push zero or many items — one per matched block/event, not one per page. bufferAtCap compares that item count to a cap.

cap = max( maxBufferedPages ?? 4 × stepSize, stepSize + 1 )
default multiplier: 4× floor: always ≥ one page + 1 
// page-helpers.ts
function bufferCapFor(syncProtocol) {
  const step = syncProtocol.stepSize ?? DEFAULT_STEP_SIZE;       // 1000 fallback
  return Math.max(syncProtocol.maxBufferedPages ?? 4 * step, step + 1);
}

At the cap, the chain's stateToInput returns undefined for that poll — the fetch loop just sleeps until the merge drains enough items.

The deadlock #

The merge only drains a chain's buffer after its page passes τ. If the cap pauses the fetcher first, nothing can move.

The fix — mergeWaitingForPage #

While the merge is gated on a chain's page, that chain is exempt from the cap. The fetch keeps going until the page passes τ.

state.mergeWaitingForPage = true;          // set by mergeIntoRoot, just before it blocks

const atCap = !state.mergeWaitingForPage && state.bufferedData.size() >= cap;

Once the page passes τ, the merge drains the buffer and the cap re-engages — the overshoot was bounded by exactly (lastPage, τ], never unbounded.
