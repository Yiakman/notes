How the sync loop fails

Two error classes, one bounded retry — the failure design of Effectstream's per-chain fetch loop (startSync), plus how transient Postgres errors are kept from crashing the node. July 2026.

Every configured chain gets one startSync loop (an effection task). Per iteration it computes the next page range, fetches it from the chain RPC, and buffers the result for the merge loop. The design question this note answers: when a stage throws, who catches it, and should the node survive?

#The five stages and their error classes

StageWhat it doesOn error
0 · startAsyncBackground producer spawn — websocket subscriptions for utxorpc/Midnight; a no-op for NTP/EVM.bounded retry 5 attempts, then rethrow → fatal
1 · stateToInputCompute the next page range to fetch (talks to the chain RPC for the tip).tolerated forever count, log, sleep, retry

3 · updateStatePush outputs to the in-memory buffer, wake the merge's condition variables.fatal escapes the task, halts the scope
4 · producerChannel.sendHand outputs to a channel…that nothing consumes.can't fail effection channels drop sends with no subscriber

The principle: stages that talk to the outside world get tolerance; in-process plumbing gets none. A chain RPC returning 503 is weather — expected, transient, survivable. A throw from updateState (pure Deque + CondVar bookkeeping) is a logic bug, and limping past it risks silent state divergence, which is strictly worse than a crash. So it escapes the spawned task, effection's structured concurrency halts the whole scope, and the process exits for the supervisor to restart.

tryYield

A ~10-line try/catch that turns an Operation's throw into a {data, error} result. Stages 1–2 are wrapped in it; that's the entire tolerance mechanism.

consecutiveErrors

Per-chain failure streak, surfaced in the node's 60-second heartbeat. Bumped on every caught error alongside lastErrorTimestamp.

Halt = crash-only recovery

An escaped error reaches effection's main(), which exits the process. Restart flows through the block-accurate resume path — the best-tested recovery code in the system.

#Why stage 0 gets a bounded retry, not infinite tolerance

Stage 0 used to be unwrapped: any throw from startAsync was instantly fatal. That violated the principle above — connecting a websocket is talking to the outside world, and a Cardano node being briefly unreachable at boot shouldn't kill the whole process.

But copying stages 1–2's retry-forever would be worse than the crash it replaces. If the producer is permanently dead, its chain's data never arrives, and the merge loop — which waits on the slowest chain before producing each block — stalls behind it indefinitely. The node looks healthy while producing nothing: a zombie. A crash at least gets a supervisor restart and an alert.

The failure asymmetry: a dead fetch loop keeps erroring visibly (streak climbs, heartbeat shows it). A dead producer fails once and then goes silent — there is no per-iteration signal left to escalate on. Bounded retry is the compromise: absorb blips, refuse to become a zombie.

So stage 0 now runs the same treatment as stages 1–2 (tryYield, streak bump, log, sleep one polling interval) but capped at 5 attempts, after which it rethrows the original error — not a generic "max attempts reached" — so the crash log at 3am names the real cause. One honest scope limit: this covers connect-time failures. A websocket that drops mid-stream, after startAsync resolved, never passes through this code again; reconnection there remains each fetcher's internal job.

#The consecutiveErrors trap: where to put the reset

The streak used to reset in exactly one place: after a successful fetch. That has a stale-gauge bug — a chain that errors a few times and then catches up (its stateToInput starts returning "nothing to fetch") never fetches again while idle, so the heartbeat reports a non-zero failure streak forever on a perfectly healthy chain.

The tempting fix is wrong in the opposite direction: reset after any successful stateToInput. Then a chain whose readData persistently fails oscillates 0→1→0→1 — the streak can never exceed 1, and the heartbeat's main signal for a broken RPC is destroyed.

The invariant that resolves both: the streak ends only when a full iteration succeeds — and there are exactly two ways an iteration succeeds: a fetch returned data, or the chain reported caught-up. Reset in both places, nowhere else.

A unit test pins each side: one asserts the streak is zero after errors-then-idle (fails on the old code), and a pre-existing one asserts the streak reaches ≥2 under persistent readData failure (fails under the naive fix).

#The other half: transient Postgres errors

Chain RPC errors are caught in-band by tryYield, in generator code, inside the loop that made the call. Postgres errors are messier: on serverless Postgres (Neon), cold-start RSTs, idle-socket evictions, and TLS blips are routine, and a dying socket can reject a promise nobody is currently awaiting — surfacing out-of-band as a process-level unhandledRejection. So pg gets its own three-layer treatment, all keyed on one classifier:

isTransientPgError

Deliberately narrow allow-list: known message fragments ("Connection terminated unexpectedly", "Query read timeout", …), or ECONNRESET/EPIPE/ECONNREFUSED with a stack inside pg/pg-pool. Walks the cause chain. Anything else — auth failures, SQL bugs — is not transient and still crashes.

Layer 1: per-block retry

processFinalizedBlockWithRetry re-runs the whole block from BEGIN on a fresh pool client (the old socket is destroyed, never recycled), with exponential backoff capped at 5s. Safe because the failed transaction never committed and the block stream only advances on success.

Layer 2: outage escalation

PoolErrorTracker aggregates consecutive transient failures process-wide. Individually they log as WARN; past 60s of sustained failure they escalate to ERROR; past 5 minutes the process exits — a real outage, hand it to the supervisor.

Layer 3: the global net

The process-level unhandledRejection handler asks the same classifier: transient pg → log WARN and survive (the pool recovers on the next query); anything else → log ERROR and exit(1). Silent bug-swallowing is never an option.

Note the contrast with the sync loop: the fetch loop tolerates chain errors forever (a chain being down for a day is the chain's problem; the node keeps serving the others), while pg failures are on a clock — the database being down means no blocks are committing at all, so past 5 minutes a restart is more likely to help than more retries.

#Where an error is thrown decides who catches it — not its shape

The two systems never overlap, and it's worth being precise about why. A pg-shaped error (right message, right code) thrown from inside a chain fetcher is caught by tryYield like any other fetch error — it bumps the streak and never reaches the global handler. The pg exemption only applies to rejections that leak out of everything, which in practice means out-of-band socket failures. The classifier's narrowness plus this structural separation is what keeps "survive transient pg errors" from quietly becoming "survive arbitrary bugs that happen to mention connections".

One suite proves each direction: the sync tests assert fetcher errors (including deliberately pg-shaped ones) never fire the global handler; the db tests assert genuine out-of-band pool rejections do — and survive only when the classifier says so.
